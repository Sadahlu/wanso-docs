## 二、工作成果(按主题)

### 主题一:性能优化 —— 推荐链路 N+1 查询根治

**提交**:`5600f67 refactor: 提取 chat 查询到 queries/ 包并批量化 N+1`(06-25)

**问题**:餐厅推荐的核心候选筛选路径存在严重的 N+1 查询问题 —— 每次用户提问,后端要在循环里对每个候选餐厅单独查询菜品匹配和相关性,单次请求最高触发 **46 次数据库往返**。

**做了什么**:
- 新建 `queries/` 包,按职责拆为 `dishes`(菜品)/ `relevance`(相关性)/ `candidates`(候选筛选)三个模块,共 569 行
- 把候选搜索改写为 **2 条批量 SQL**(1 条取餐厅 + 1 条取菜品),用 `CASE WHEN` 给关键词排序,彻底消除循环查询
- 预算过滤、类目 gate、`get_best_dish`、`calculate_menu_relevance` 全部改批量预取
- 从 `server.py` 净移出 357 行(−461 / +104),代码同时变薄

**业务影响**:**/api/chat 响应延迟显著下降,数据库压力大幅降低**,直接提升用户提问时的等待体验和系统并发承载能力。

### 主题二:用户体验 —— 推荐开场白显示修复

**提交**:`79e76af refactor: 移除 chat greeting 的 150 字符截断`(06-24)

**问题**:推荐结果的开场白(greeting)在流式与非流式两条路径上都被硬截断到 150 字符,而该截断会**在多字节中文字符中间生硬切断**,产出乱码式断句。

**做了什么**:删除 `_stream_recommend_ai` 中的 `GREETING_LIMIT` 硬截断和非流式断句逻辑,长度改由 AI prompt 控制(让模型自己输出合适长度)。

**业务影响**:**中文/日文用户不再看到被生硬切断的开场白**,推荐开场更自然。

### 主题五:推荐稳定化设计 —— follow-up 推荐链路重构(与同事协作)

**协作提交**:`19a0c39 fix: stabilize follow-up restaurant recommendations`(经 develop 分支流转,06-28 通过 `4dacfba` 合入 dev-gbz;提交身份为团队 backend 账号,设计由我与同事共同讨论落地)

**问题**:用户在推荐结果上追问"还有吗 / 换一家"时,后端 follow-up 路径存在多处稳定性缺陷 —— 会把已关店或被删除的餐厅塞回客户端、推荐开场白里价格被 AI 误用汇率转换、菜单匹配在多语言和带重音字符时漏匹配、fastpath 与主路径混在一起难以按需切换。

**做了什么**(对应 37 条 `WANSO_*` 版本标记,详见 `backend/analysis/followup-stabilization-19a0c39.md`):
- **统一最终加工守卫层** `_wanso_finalize_chat_response_for_app`(V1→V8):在响应返回客户端前统一执行过滤 + 字段归一化 + 候选补全,所有 chat 出口走同一份契约
- **营业/删除状态过滤**(V1):不再把已关店或被删除的餐厅返回给客户端
- **"更多餐厅"追问流式补全**(V14H / V14P2):用户问"还有吗"时,从菜单搜索中补足仍在营业的新候选并接入 SSE stream
- **卡片字段归一化 + VND-only 价格**(V2 / V14P2):价格统一成整数 VND,移除多币种转换(注释明确:"AI tends to misuse conversion in greeting")
- **严格菜单 + 菜系匹配**(V4/V5/V7):多语言、去重音的菜单匹配器,替代旧的 `term_matches` 路径
- ⭐ **fastpath 可配置化**(V20,`WANSO_FOOD_FASTPATH_ENV_SWITCH_V20`):新增环境变量 `WANSO_FOOD_FASTPATH_MODE`,默认 `"on"`,**支持运行时一键开关 fastpath 路径** —— 这是我与同事讨论后落地的设计,目的是把 fastpath 与主路径解耦,线上出问题时无需改代码重新发版,改 env 即可降级回主路径,显著降低回滚成本和事故响应时间

**业务影响**:
- **追问场景稳定性提升**:关店餐厅不再误推、追问补全不再空手而归
- **运维灵活性提升**:fastpath 可配置化让推荐主链路具备"灰度切换/快速降级"能力,**事故响应从"改代码重新发版"压缩到"改 env 重启"**
- **AI 误用率下降**:VND-only 价格消除 AI 在 greeting 里乱换汇率的坑

### 主题六:可观测性 —— /api/chat 完整调用链 trace 日志(本周新增)

**状态**:本地工作(`chat_tracelog.py` + `server.py` 接入,待提交)

**问题**:`/api/chat` 推荐链路有 11 个阶段(限流 / GPS 默认 / geocode / 响应缓存 / 语言检测 / 会话状态 / 严格食物词短路 / AI 意图 / 候选筛选 / AI 生成 / SSE 流式),但此前**没有任何机制能追踪一次请求实际走了哪条路径、各阶段耗时多少、AI 调用花了多少 token、候选筛选命中多少**。出问题时只能靠散落在代码各处的 print 拼凑,排查一次用户报告的"推荐不对"往往要翻几小时日志。

**做了什么**:
- 新建 `chat_tracelog.py`(209 行),基于 `contextvars` 实现**按 `req_id` 串联的完整调用链追踪**,输出到 `logs/chat_trace_YYYYMMDD.jsonl`
- 在 `server.py` 接入 **17 个 trace 点**,覆盖 chat 全链路关键决策点:
  - **请求入口 + fastpath 短路点**:constraint / rate_limited / chat_cache_30min / geocode_hit
  - **AI 意图分类**:intent + food_keywords + fastpath 路径标签
  - **关键词阶段转换**:seed/expand 前后对比
  - **候选 DB 查询**:label + hits + latency_ms + sql_hint
  - **排序阶段**:input/output count + 策略
  - **每次 AI 调用**:model + latency_ms + prompt/completion/cached tokens + resp_preview
  - **最终 AI 响应**:picks_count + greeting/followup 长度
- **stage 上下文管理器**通过 contextvar 经 `asyncio.to_thread` 传播到工作线程,确保 17 个 gunicorn worker 下 trace 不串号
- **容错原则**:任何日志失败静默吞错,绝不影响业务;非请求上下文(`req_id is None`)跳过,**不污染其他模块的 AI 调用日志**
- 提供语义化 helper(`log_intent` / `log_db` / `log_ai` / `log_ranking` 等),字段名固定,**便于 jq 过滤**

**业务影响**:
- ⭐ **排障时间大幅下降**:事后一条命令即可重建单次请求的完整链路
  ```bash
  jq 'select(.req_id=="HHMMSS-xxxxxx")' logs/chat_trace_*.jsonl
  ```
  从几小时翻日志压缩到几秒
- **性能优化有据可依**:每条 AI 调用都有 `latency_ms` 和 token 计数,后续做 prompt 优化、模型选型、缓存策略调整都有数据支撑
- **测试与回归可验证**:每次改动后跑典型 query,对比 trace 即可看出哪个阶段行为变了
- **为后续 A/B 测试和 SLO 监控打基础**:trace 是结构化 JSONL,可直接喂给后续的 analytics pipeline
